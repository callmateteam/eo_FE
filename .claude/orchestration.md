# 총괄 셰프

## 기본 흐름

```
요청 파악 → 관련 파일 확인 → 레시피 선택 → 구현 → 검증 → (필요시) 기록
```

1. 최종 결과를 먼저 파악한다.
2. 관련 파일을 직접 읽어서 확인한다. 추측하지 않는다.
3. [skills.md](skills.md)에서 가장 작은 레시피를 고른다.
4. 기존 구조를 유지하면서 최소 범위로 수정한다.
5. 작업 규모에 맞는 가장 가벼운 검증을 실행한다.
6. 이후에도 중요할 결정이면 [memory.md](memory.md)에 기록한다.

## 검증 기준

| 작업 규모 | 검증 방법 |
|-----------|-----------|
| 작은 시각 수정 | 시각 확인 또는 영향 경로 확인 |
| 여러 파일 수정 | `npm run lint` + 타입 오류 확인 |
| 구조·타입 변경 | 위 + `npm run build` |

## 작업별 분기

**UI 변경** — 대상 페이지와 공용 컴포넌트 먼저 확인. 기존 패턴 재사용 우선.

**버그 수정** — 증상이 아닌 근본 원인을 수정. 실패 경로를 코드에서 직접 추적.

**리팩터링** — 의미 있는 중복이 두 곳 이상일 때만 추상화. 기존 동작 유지 최우선.

**리뷰** — 발견 사항을 심각도 순으로 정리. 영향 파일과 실제 리스크 함께 명시.

## 완료 기준

- 요청된 변경이 코드베이스에 반영됨
- 적절한 검증을 실행했거나, 못 한 이유가 명확히 적힘
- 지속적 관례가 생겼다면 [memory.md](memory.md)에 기록됨
