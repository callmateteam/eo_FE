# Orchestration

이 파일은 Codex 작업 흐름에서 "총괄 셰프" 역할을 담당합니다.
무엇을 먼저 확인할지, 어디를 수정할지, 어떻게 검증할지를 여기서 정합니다.

## 기본 흐름

1. 사용자에게 필요한 최종 결과를 먼저 식별합니다.
2. 제안하거나 수정하기 전에 정확히 관련된 파일을 확인합니다.
3. [skills.md](/C:/Users/USER/Desktop/coding/eo/skills.md)에서 가장 작은 적합한 레시피를 고릅니다.
4. 영향 범위를 최소화하는 방식으로 구현합니다.
5. 작업 규모에 맞는 가장 가벼운 검증 명령을 실행합니다.
6. 나중에도 중요할 맥락이면 [memory.md](/C:/Users/USER/Desktop/coding/eo/memory.md)에 기록합니다.

## 실행 원칙

- 추측보다 직접적인 코드 확인을 우선합니다.
- 새로운 평행 구조를 만들기보다 기존 구조를 수정하는 편을 우선합니다.
- 변경 범위가 넓지 않다면 전체 프로젝트 검사보다 타깃 검증을 우선합니다.
- 작은 시각 수정은 시각 확인이나 영향 경로 확인처럼 가장 가벼운 검증을 우선합니다.
- 여러 파일을 수정했거나 타입 경계가 바뀌면 타입 오류 확인을 수행합니다.
- 구조 변경이나 복잡한 작업 후에는 가능하면 빌드 확인까지 수행합니다.
- 워크트리가 이미 더럽더라도 관련 없는 변경은 되돌리지 않습니다.

## 작업 유형별 분기

### UI 변경

- 대상 페이지, 관련 컴포넌트, 공용 스타일을 먼저 확인합니다.
- 가능하면 기존 간격, 타이포그래피, 컴포넌트 패턴을 재사용합니다.
- 가능하면 시각적으로 검증하고, 여러 파일 수정이나 복잡한 변경이 포함되면 `lint`, 타입 오류 확인, 제한된 빌드 확인 중 맞는 범위를 실행합니다.

### 버그 수정

- 로컬 코드 기준으로 실패 경로를 재현하거나 추론합니다.
- 증상만 막지 말고 근본 원인을 수정합니다.
- 재발 방지에 도움이 된다면 조건문, 가드, 상태 전환을 더 명확하게 만듭니다.
- 공용 코드나 여러 파일에 영향이 있으면 `lint`와 타입 오류 확인을 우선하고, 구조 영향이 크면 빌드 확인도 검토합니다.

### 리팩터링

- 먼저 기존 동작을 유지합니다.
- 반복 로직이나 반복 리스크를 줄일 수 있을 때만 추상화를 도입합니다.
- 유지보수성이 실질적으로 좋아지지 않는다면 광범위한 이름 변경은 피합니다.
- 외부 인터페이스나 타입 구조가 바뀌면 타입 오류 확인을 수행하고, 런타임 영향이 크면 빌드 확인도 검토합니다.

### 리뷰

- 발견 사항을 먼저 제시합니다.
- 심각도 순서로 정리합니다.
- 영향 파일과 실제 리스크를 함께 적습니다.

## 완료 기준

아래 조건이 충족되면 작업이 완료된 것으로 봅니다.

- 관련 검증을 실행했거나, 실행하지 못한 이유가 명확히 적힘
- 작업 중 발견한 지속적 관례가 있다면 [memory.md](/C:/Users/USER/Desktop/coding/eo/memory.md)에 기록됨
